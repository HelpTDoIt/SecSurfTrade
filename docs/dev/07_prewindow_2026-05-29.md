### Cost-optimized execution — revised 2026-05-29 PM

User constraints: ~2.5–3 hr trading window today, ~50% of daily Pro plan quota remaining. Plan re-scoped for lowest token cost that still fits.

**Scope cut.** `localStorage` persistence is removed from today's WS-D. Only the `?import=<path>` handler ships today (≈20 LOC) — that's what's required for M1's one-command morning to work end-to-end. Persistence moves to **next-window priority** in the Under Consideration roadmap.

**Mitigation for deferred persistence.** On Monday, click **⬇ Export State** between rounds for a manual snapshot. If the browser dies mid-window, re-import the most recent export and re-enter fills since that point.

**Execution model.**

| Phase                       | Items                                   | How                                                                        |
| --------------------------- | --------------------------------------- | -------------------------------------------------------------------------- |
| Phase 1 — inline build      | WS-B1, WS-B2, WS-B3, WS-E               | Main thread, sequential, no subagents (small new files, already-read code) |
| Phase 2 — one subagent      | WS-D-mini (`?import=` handler only)     | Single Sonnet subagent against main checkout                               |
| Phase 3 — live verification | T1, T4, then T2, T5, T6, T3, T7, U1, M1 | User runs commands; main thread reviews short outputs                      |

**Why inline beats parallel subagents here.** Subagent overhead (system-prompt re-load, file re-reads, fresh context tax) exceeds the wall-clock parallelism savings for tasks under ~150 LOC. WS-D is the only build item large enough that a Sonnet subagent (cheaper rate × isolated context for 1,292-line HTML) wins on total tokens.

## Selection rationale

**U1 picked over alternatives** (keyboard nav, copy-as-ticket, live spread strip, dark mode): losing state mid-window is the only _catastrophic_ failure mode in section 2 — the rest are incremental UX. Persisting `localStorage` is ~50 lines of focused JS and survives accidental reloads, browser crashes, and lost network. It also unblocks tomorrow's improvements (keyboard nav etc. all benefit from durable state).

**M1 picked over alternatives** (filesystem watcher, account-selector write adapter, auto-advance round, OCR fill auto-log): all of the OCR / UIA-write items require careful adapter work and live-window validation. M1 is pure shell orchestration of already-validated CLIs — lowest risk, lowest effort, runs every single trading day for the foreseeable future. The harder reducers (OCR fill logging, fs watcher) will be much easier to add _after_ M1 standardizes the morning entry point.

```
   │ T3  Ex-div dry-run (runs WS-B2 output)                       │

   │ U1  Browser smoke of ?import= handler                        │
   │ M1  End-to-end dry run of morning-prep.ps1                   │
   └──────────────────────────────────────────────────────────────┘
```

### Why this shape

- **Phase 1 inline.** WS-B1/B2/B3/E are each <150 LOC of new code touching files already in main-thread context. Spawning subagents would pay system-prompt and file-re-read overhead that exceeds the parallelism savings. Sequential execution in the main thread is the lowest-cost shape.
- **Phase 2 one subagent.** `rebalance_calculator.html` is 1,292 lines. A Sonnet subagent (cheaper per-token than Opus main thread) with a narrow brief reads it once in an isolated context and ships a focused patch — better total cost than reading it into main-thread Opus context.
- **Phase 3 user-driven.** Verification steps are mostly the user running CLIs and reporting pass/fail. Main-thread cost is just short-output review, which is cheap.
